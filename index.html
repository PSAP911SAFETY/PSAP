Emergency 911 & CMAS Alert Systems — Deep Technical Dive

How cellular networks route emergency calls, determine caller location, and broadcast mass-alert messages — from EPS architecture through 5G NR, NG911, and Wireless Emergency Alerts.

3GPP TS 23.167 3GPP TS 22.268 ATIS J-STD-101 NG911 / i3 IMS / E-CSCF

Overview

Emergency communications over cellular networks involve two entirely different but complementary systems: E911 (Enhanced 911) for person-to-dispatcher voice calls, and CMAS/WEA (Commercial Mobile Alert System / Wireless Emergency Alerts) for network-to-subscriber broadcast warnings. Both systems have evolved from 2G circuit-switched origins to fully IP-based architectures in LTE and 5G NR.

This article breaks down both systems — their architecture, signaling flows, protocol stacks, location technologies, and the key 3GPP specifications that define them. We also cover the transition to NG911 and enhancements introduced in 5G for both emergency calling and alert delivery.

Part 1 — Enhanced 911 (E911)

What is E911?

E911 is the US standard (mandated by the FCC under 47 CFR Part 20) for routing mobile emergency calls to the correct PSAP (Public Safety Answering Point) along with the caller's ANI (Automatic Number Identification) and ALI (Automatic Location Identification). Equivalent standards exist globally: eCall (EU), 112 (ITU-T), and 999 (UK).

Phase I E911 delivered callback number and cell tower. Phase II (2002) required precise GPS/network lat/lon coordinates. The ongoing NG911 transition adds IP-based PSAPs, dispatchable floor-level location, and multimedia support.

Emergency Call Detection (Priority Order)

01
EF_ECC — SIM/USIM Emergency Call Codes
USIM Elementary File · Highest Priority
The USIM stores up to 5 operator-defined emergency numbers in EF_ECC (3GPP TS 31.102). These override all other sources and are used when a valid SIM is present.
02
SIB1 / SIB2 Emergency Number List
Broadcast from serving cell · Medium Priority
The serving cell broadcasts local emergency numbers. In LTE: SIB10/11; in 5G NR: SIB2 → emergencyNumberList. Allows networks to inform UEs of country-specific numbers.
03
UE Hardcoded List
Device firmware · Fallback (incl. no-SIM)
Every UE maintains hardcoded universal emergency numbers: 911, 112, 999, 000, 110. This list activates when no SIM is present or EF_ECC provides no data.
⚠️
No-SIM Emergency AccessPer 3GPP TS 22.101 §10.3 and FCC 47 CFR §20.18, all cellular devices must connect emergency calls regardless of SIM. With no SIM: identity = IMEI/PEI only, authentication skipped, NULL security applied, any PLMN must accept. Callback not possible (no MSISDN).

LTE E911 Signaling Flow

01
Emergency Detect + Cell Selection
UE (local)
UE matches digits against EF_ECC/SIB/hardcoded list. If not registered, initiates Emergency Attach on any available PLMN.
02
Emergency PDN + QCI-1 Bearer (ARP=1)
MME → S-GW → P-GW
Emergency PDN to IMS Emergency APN. QCI=1 GBR bearer with ARP Priority Level=1, preemption capability enabled — can displace any bearer for radio resources.
03
SIP INVITE (urn:service:sos)
UE → P-CSCF → E-CSCF
P-CSCF detects urn:service:sos (RFC 5031) + Priority: emergency header. Routes direct to E-CSCF, bypassing S-CSCF and IMS registration requirements.
04
Location via LRF/E-SMLC (LPP)
E-CSCF → LRF → E-SMLC → UE
E-CSCF triggers location via LRF. E-SMLC initiates LPP with UE. Best method used: A-GNSS preferred, OTDOA/E-CID fallback.
05
PIDF-LO → E-CSCF → ESRP → PSAP
LRF → E-CSCF → ESRP → PSAP
Location returned as PIDF-LO (RFC 4119). ESRP uses LoST (RFC 5222) to select the correct PSAP. INVITE includes Geolocation SIP header with PIDF-LO.
06
PSAP Answers + RTP Voice
PSAP → UE (bidirectional RTP)
CAD shows MSISDN and location. Voice via QCI-1 bearer. Location may be re-queried periodically for mobile callers.
07
Post-Call Callback Window
Network (~5 min)
Emergency registration retained ~5 min. PSAP can initiate SIP callback. Emergency PDN connection kept alive during this window.

5G NR E911 Enhancements

4G LTE Emergency
MME handles attach. E-SMLC positioning via SLs. GMLC location to E-CSCF. QCI-1 bearer S-GW/P-GW. Methods: OTDOA, A-GNSS, E-CID.
5G NR Emergency
AMF handles registration. LMF positioning via NLs. Location to E-CSCF via LMF. 5QI-1 PDU via SMF/UPF. Methods: UL-TDOA, Multi-RTT, NR-PRS, barometric.
📡
5G NR Positioning (Rel-16+) UL-TDOA, DL-TDOA, Multi-RTT, and enhanced A-GNSS. Release 17 added barometric sensor integration for floor-level (z-axis). Release 18 introduced sidelink positioning for D2D emergency scenarios (TS 38.305).

Next Generation 911 (NG911)

NG911 modernizes 911 from legacy TDM/PSTN to an IP-based ESInet using the NENA i3 architecture (NENA-STA-010). The ESInet carries SIP-based calls, text, images, and video alongside voice.

Part 2 — CMAS / Wireless Emergency Alerts (WEA)

What is CMAS?

The Commercial Mobile Alert System (CMAS) — branded as Wireless Emergency Alerts (WEA) — allows authorities to broadcast emergency alerts to all capable mobile devices in a geographic area. It operates over 3GPP Cell Broadcast Service (CBS): a one-way, network-to-device broadcast requiring no UE registration or per-device signaling.

Alert Categories

Presidential
🔴 Presidential Alert
Issued only by the President for national emergencies. Cannot be opted out. Only two national tests conducted (2018, 2019).

Imminent Threat
🔶 Extreme / Severe Threat
Tornadoes, tsunamis, flash floods, severe storms, nuclear hazards. Extreme (immediate) and Severe (prepare now) sub-types. User opt-out allowed.

Child Abduction
🔵 AMBER Alert
Confirmed child abductions meeting AMBER criteria. Contains vehicle/suspect info. Users may opt out. Distinctive audio tone.

Test / Exercise
🟢 RMT / Operator Test
Required Monthly Test (RMT) and Required Weekly Test (RWT) to verify end-to-end CMAS infrastructure. Users may opt out.
IDs: 4355–4356 (0x1103–0x1104)

CMAS Architecture & Flow

01
Alert Originator Issues CAP Message
Federal / State / Local Authority → FEMA IPAWS-OPEN
Authorized originators submit alerts using CAP (Common Alerting Protocol, OASIS CAP v1.2). CAP is an XML standard that includes message text, affected area polygon, severity, urgency, and expiry time.
02
CAP → CMAM Translation at FAG
FEMA IPAWS → Federal Alert Gateway (FAG)
FAG translates CAP into a CMAM (Commercial Mobile Alert Message) per ATIS J-STD-101. Validates originator authorization, deduplicates, and applies the target geographic polygon.
03
CMAM Pushed to Carrier CMGs
FAG → Carrier CMAS Gateways (CMG)
FAG simultaneously pushes to all carrier CMGs via HTTPS. Delivery must occur within 10 minutes of originator submission. CMG is the carrier's interface between FEMA and their broadcast infrastructure.
04
CBC Generates Cell Broadcast Message
CMG → CBC (Cell Broadcast Centre)
CBC translates CMAM into a 3GPP CBS message. Determines target cells (polygon-to-cell mapping using cell sector footprints). Generates the CBS broadcast schedule.
05
Write-Replace Warning: CBC → eNB/gNB
CBC → MME/AMF → eNB/gNB (SBc-AP / NGAP)
CBC sends Write-Replace Warning Request to MME (LTE via SBc) or AMF (5G via NGAP). MME/AMF forwards to target eNBs/gNBs with cell list, repetition period, and message content.
06
Over-the-Air CBS Broadcast
eNB/gNB → All UEs (no registration needed)
Broadcast on BCCH using SIB12 (LTE) or SIB6/7/8 (5G NR). Repeated at specified interval until expiry. No UE-to-network signaling — UE passively receives.
07
UE Decodes & Displays Alert
UE (no response to network)
UE decodes CBS message, checks Message ID against CMAS class table, displays full-screen alert with WEA tone and vibration if class is enabled (based on user opt-out settings).

Analyzing E911 Logs from the Device

When debugging E911 call failures or validating correct emergency behavior, device‑side logs are invaluable. They reveal exactly how the UE detects an emergency number, which radio and core procedures it triggers, and whether location information is properly retrieved. Below we cover common log sources, key events to look for, and sample log snippets.

Log Sources

  • Android Bugreport – Contains logcat (IMS stack, RIL, telephony), kernel logs, and event logs. adb bugreport captures everything.
  • QXDM / QCAT (Qualcomm devices) – Low‑level modem logs showing NAS, RRC, and SIP messages between UE and network.
  • Wireshark + tethering – When the UE is used as a modem, a tethered PC can capture IP traffic (SIP/RTP) for analysis.
  • iPhone Field Test / sysdiagnose – Apple’s logs contain similar NAS/IMS information (accessible via privacy logs).
  • Vendor diagnostic tools – MediaTek (META), Samsung (Service Mode), etc.

Key Events to Look For

1
Emergency number detectionLogs show “Emergency number: 911” or “ECCM” (Emergency Call Codes Match).
2
Emergency attach / registrationNAS layer indicates “Attach type: EPS emergency attach” or “Registration type: Emergency”.
3
PDN / PDU session with emergency APNAPN = “sos” or “emergency” ; QoS = QCI‑1/5QI‑1, ARP=1.
4
SIP INVITE with sos URNIMS logs show INVITE sip:urn:service:sos and Priority: emergency header.
5
Location retrieval (LPP / RRLP)LPP messages (RequestLocationInformation / ProvideLocationInformation) with A‑GNSS or OTDOA data.
6
PIDF‑LO generationSIP Geolocation header containing presence XML with coordinates.

Tools & Tips

  • logcat – Filter by Emergency, ECCM, RILJ, IMS, LPP.
  • QXDM/QCAT – Look for NAS EMM messages (Attach Request/Reject, PDN Connectivity) and RRC messages (RRCConnectionRequest with establishmentCause = emergency).
  • Wireshark – On tethered mode, capture SIP traffic; filter sip and look for urn:service:sos in the Request‑URI.
  • Cross‑reference with call flows – Match the logged steps against the diagrams on this site to identify missing or rejected messages.
📱
Debugging hint – On Android, adb logcat -b all -d > bugreport.txt captures all buffers. Use grep -i emergency to quickly locate emergency‑related entries.
